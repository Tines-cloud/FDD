## THE #1 CAUSE OF COMPILATION FAILURE: Nested dot paths

The errors `Found "." expecting ";"` and `Found "." expecting "("` BOTH mean you
wrote a multi-level dotted path where only ONE level is allowed.

**SOURCE side** (causes `Found "." expecting ";"`):
  src.name.family as f -> ...           <- WILL NOT COMPILE
  src.address.line as l -> ...          <- WILL NOT COMPILE
  src.result.reference as r -> ...      <- WILL NOT COMPILE

**TARGET side** (also causes `Found "." expecting ";"`):
  src -> tgt.coding.system = "url" "r"; <- WILL NOT COMPILE
  src -> tgt.category.text = "x" "r";  <- WILL NOT COMPILE

FML does NOT support chained dots beyond one level on EITHER side. Use sub-groups.
  CORRECT:  src.name as sn -> tgt.name as tn then MapName(sn, tn) "map-name";
  Then define group MapName with: src.family as f -> tgt.family = f "set-family";

If you see MANY `Found "." expecting "("` errors, find EVERY multi-level dotted
path in your code and replace them with sub-group delegation.

### C. Copying a field two or more levels deep (NEVER use src.a.b inline)

### I. Multi-line conditional "then" construct - NEVER valid FML
  ERROR:    Found "." expecting ";"  (on the source line) + cascade of "expecting group" errors

## CRITICAL: No oscillation or regression

When you receive follow-up errors, it means your PREVIOUS fix attempt introduced
those errors. You MUST fix the current errors WITHOUT reverting to any prior version
of the code that you already know was broken. Build forward incrementally - do not
undo your prior corrections. If you see "PREVIOUSLY SEEN ERRORS", those are errors
from earlier versions that you must NOT reintroduce.

If you see "REGRESSION DETECTED", it means your fix for a specific line replaced
one error with a DIFFERENT error on the SAME line. Both your old and new versions
of that line are wrong. You MUST pick a DIFFERENT pattern letter (A/B/C/D/E above)
for that line - not a minor variation of the same approach.

If you see "OSCILLATION DETECTED", the same exact error has appeared in 2+ cycles.
Do NOT try yet another variation of the same construct. Switch to pattern E
(comment + omit) if no other pattern can express the mapping cleanly. A FML that
compiles but skips one drift item is far more useful than one that fails to compile.

Your fix MUST NOT introduce any new compilation errors. If you are unsure about
the correct syntax, use a simpler construct that you know is valid.

## "Complex rules must have an explicit name"

This is the #2 most common error. HAPI-FHIR requires EVERY rule (even simple ones)
to have a quoted name string at the end, before the semicolon. The name must be a
unique string literal in double quotes.

CORRECT examples:
  src.id as v -> tgt.id = v "set-id";
  src.active as v -> tgt.active = v "set-active";
  src.name as srcName -> tgt.name as tgtName then HumanName(srcName, tgtName) "name";
  src.extension as ext where ext.url = 'http://example.org/ext' -> tgt.extension as tgtExt then MapExt(ext, tgtExt) "map-ext";
  src -> tgt.status = 'active' "default-status";

INCORRECT examples:
  src.id as v -> tgt.id = v;                   <- MISSING quoted name
  src.name as srcName -> tgt.name as tgtName then HumanName(srcName, tgtName);  <- MISSING quoted name

## Other common errors and fixes

- Missing semicolons -> add ';' at the end of EVERY rule, after the quoted name
- Incorrect group signatures -> use: group Name(source src : Type, target tgt : Type)
- Invalid 'uses' declarations -> The ONLY valid modes are 'source' and 'target'.
  CORRECT: uses "url" as source   OR   uses "url" as target
  WRONG: uses "url" alias Name as source    <- 'alias' is not valid
  WRONG: uses "url" as conceptMapUrl         <- not a valid mode
- Missing or malformed map declaration -> must be: map "url" = "name"
- "Unknown StructureMapInputMode code 'X'" -> the 'uses' line has an invalid mode.
- Do NOT use dotted types like 'Patient.Contact' as group parameter types.
  Use 'BackboneElement' instead.
